# 光构部署总览

[English](../../en/deployment/index.md) · [文档中心](../../README.md)

本文档是生产部署入口。光构支持两种部署方式，两种方式的目录结构、进程管理和升级命令不同，请先选择一种，然后只执行对应文档中的步骤，不要把两套命令混用。

## 选择部署方式

| 方式 | 适合谁 | 优点 | 需要自行负责 |
| --- | --- | --- | --- |
| [Docker Compose 部署](./docker.md) | 首次部署、单机生产环境、希望快速上线的用户 | 环境一致，MySQL、Redis、前后端一次启动，升级和迁移更直观 | Docker、域名、HTTPS、备份和服务器安全 |
| [传统服务器手动部署](./manual.md) | 熟悉 Linux、systemd 和 Nginx，需要精细控制进程与目录的用户 | 服务目录、日志、数据库和反向代理都可独立管理 | Node.js、Go、MySQL、Redis、Nginx、systemd 和版本升级 |

如果没有明确的运维偏好，建议使用 Docker Compose。手动部署不是“更简单的 Docker”，它要求你自行维护每个系统组件。

## 两种方式共同的生产要求

无论选择哪种方式，上线前都必须准备以下内容：

1. 一台 Linux 服务器。建议至少 2 核 CPU、4 GB 内存，并为数据库、构建缓存和用户图片预留足够磁盘。
2. 一个已经解析到服务器公网 IP 的域名。
3. 仅对公网开放 80 和 443 端口。MySQL、Redis 和后端 3000 端口不能直接暴露。
4. 两个不同的高熵密钥：`SESSION_SECRET` 和 `USER_TOKEN_SECRET`，每个至少 32 字节。
5. 一个可用的 OpenAI 兼容图片上游、API Key 和实际模型名称。
6. 数据库与 `GOUO_ASSET_DIR` 的成套备份方案。只备份数据库无法恢复用户图片，只备份图片目录也无法恢复任务关系。

推荐在服务器上生成密钥：

```bash
openssl rand -hex 32
openssl rand -hex 32
```

两次输出分别保存为 `SESSION_SECRET` 和 `USER_TOKEN_SECRET`。不要使用文档示例值，不要把真实密钥提交到 Git，也不要在截图、工单或聊天记录中公开。

## 部署后的统一配置顺序

服务启动成功并不代表可以直接开放注册。第一次上线应按以下顺序配置：

1. 通过本机访问或 SSH 隧道进入 One Hub 管理端 `/panel`。
2. 使用初始化账号登录后立即修改 `root` 的默认密码 `123456`。
3. 在“渠道”中添加上游 API Key，并配置 `gpt-image-2` 或对应的模型映射。
4. 验证图片生成、编辑和变体接口。
5. 核对固定售价、成功扣费、失败退款和余额不足流程。
6. 在“运营 → 光构存储”确认云端图片可写、用户配额正确、管理员可以查看输出作品。
7. 配置 HTTPS、备份、监控和防火墙。
8. 使用普通用户完成注册、登录、生成、同步、回收站和跨浏览器恢复测试。
9. 完成[生产上线检查清单](./checklist.md)后再开放注册。

## 重要目录和数据

| 数据 | 作用 | 是否必须备份 |
| --- | --- | --- |
| MySQL 数据库 | 用户、渠道、令牌、额度、日志、作品任务和资产关系 | 必须 |
| `GOUO_ASSET_DIR` | 用户输出图、参考图、遮罩和缩略图 | 必须，并与数据库保持同一备份时间点 |
| `SESSION_SECRET` | 登录 Cookie 签名 | 必须安全保存；丢失或更换会使现有登录失效 |
| `USER_TOKEN_SECRET` | 用户中继令牌签名 | 必须安全保存；更换后旧令牌失效 |
| Redis | 缓存和多实例协调 | 建议备份配置；它不能替代数据库 |
| 前端构建产物和后端二进制 | 当前运行版本 | 可以从同一 Git 提交重新构建，但应记录发布提交号 |

用户图片不能通过 Nginx 静态目录公开。图片必须继续由 `/api/gouo/assets/:id/content` 等鉴权接口读取。
